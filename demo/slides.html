01 · the problem

There are too many AI tools
and nobody knows which fit them.

A new AI product launches every week. Reviews are gamed. "Best of" lists rank by SEO. Founders pay for impressions on platforms that don't know who their user is.

The match between an AI tool and the user it would actually change a week for is invisible to both sides.

  • Users default to ChatGPT, Notion, Slack — household names everyone already knows.
  • Founders blow ad budgets on impressions disconnected from fit.
  • The right tool for the right person doesn't find a path.

02 · the solution — founder side

Pay once. Reach users built for you.

Founders submit a launch with URL, problem statement, ICP, and target communities. Mesh embeds the ICP, runs the inverse of the user-side matching algorithm, and surfaces the launch only to the top 5% by cosine similarity.

/01

Submit

5-step form: URL · problem · ICP · pricing · target communities. Founder presence links optional.

/02

Verify

Human admin reviews within 24h. Approval triggers the publish pipeline. No auto-approval in production.

/03

Reach

Auto-post to communities · concierge nudge to top-5 matched users · live in /p/{slug} with upvote + analytics.

CPA on qualified engagement only. No CPM. No "sponsored" placement on the organic ranking.

05 · the algorithm — i

One embedding space.
Two towers meeting in it.

Mesh borrows the two-tower retrieval idea: encode both sides of a match into the same vector space, then retrieve by cosine. Users → ProfileEmbedding. Tools → ToolEmbedding.

user tower

profile_text → vector

role · level · industry
+ daily-stack picks
+ task signature
+ friction phrasing
⇋ cosine ≥ 0.85
tool tower

name + tagline + description → vector

slug · category
labels (BM25 surface)
tagline (BM25 surface)
description (BM25 surface)

same engine matches both directions: users find tools · founders find users.

06 · the algorithm — ii

Same cosine.
Schedule shifts per question.

Each tap, the user vector grows from concrete (role, stack) to semantic (task, friction). Hybrid alpha shifts from BM25-leaning to vector-leaning to match.

Q1 · ROLE
α 0.3
70% keyword · sparse signals lead
Q2 · STACK
α 0.5
balanced · stack tokens are sparse-friendly
Q3 · TASK
α 0.7
30% keyword · task vocab is semantic
Q4 · FRICTION
α 0.8
20% keyword · friction phrasing is semantic

household-name tools (label = all_time_best) filtered out post-hybrid. niche profiles return what they return — no padding.

07 · features — what shipped

17 cycles. 15 features. End to end.

/01 · auth + role split

User vs founder, structurally

JWT (HS256, 7-day) with bcrypt; require_role middleware; constitutional invariant — founders never post in user communities.

/02 · question bank

Append-only answers + lazy profiles

Classic onboarding loop; answers immutable for audit/export. profiles lazy-create with last_invalidated_at + last_recompute_at for cache validity.

/03 · catalog + curation

Seed + admin-gated catalog

tools_seed (curated organic) vs tools_founder_launched (paid launches). Separate collections — constitutional invariant.

/04 · weaviate pipeline

Hybrid-ready vector layer

Lazy-init Weaviate v4 client · ProfileEmbedding + ToolEmbedding classes · gRPC w/ REST fallback · 8s init timeout for VPN tolerance.

/05 · fast onboarding match

Two-mode match dispatch

Generic mode under 3 answers · embedding mode after. ~50ms / ~600ms targets. Graceful degradation when embeddings unavailable.

/06 · recommendation engine

Cached + LLM-rerank organic recs

Profile vector → similarity_search → GPT-5 listwise rerank · 7-day cache · 5 picks with verdict + reasoning.

10 · repo learnings

The actual task
was fluency in the discipline.

SDD + VKF aren't documentation chores — they're a forcing function for shipping coherent product.

where it helped
  • Constitution as guardrails. Three constitutional invariants (separate collections · no founders in user communities · CPA only) prevented dozens of bad decisions across 15 cycles.
  • Spec-delta forced cross-system thinking. Every cycle had to specify backend + frontend + tests + downstream effects. No "ship the API and figure out the UI later."
  • Cycles compose. Cycle #15's live flow could ship in 2 days because cycles #4 (Weaviate), #6 (engine), #12 (inbox) were already solid abstractions.
  • learnings.yaml beats memory. 51 entries — specific, dated, reasoned. Future-me reads them, not future-me's vague recollection.
frictions
  • Spec-delta GWT scenarios slow on UI work. Given/When/Then is great for backend invariants; less great for "the graph fade-in feels right." Wrote them anyway; sometimes it was theatre.
  • VKF and SDD overlap on architecture. Where does "Weaviate is gRPC-based" live? Architecture doc · technical-constitution · feature spec · all three? Picked architecture/system-design as canonical.
  • Cycle granularity tradeoff. Cycle #13 was a 4000-line monster (frontend-core); cycle #14 was 2000 (frontend-secondary). In hindsight three cycles of 1500 each would've been cleaner reviews.
  • Validation tasks are hard to truly tick. "End-to-end smoke" is rarely fully verified mid-cycle; usually validated only at /sdd:complete time, sometimes deferred.
11 · hackathon learnings — i

Karpathy's loop, applied to everything.

The same loop ran on UX, algorithm research, repo discipline, and code. Generate → score → keep / abandon → next → synthesize.

algorithm research loop

16 iterations. Each: research agent generates a retrieval approach (semantic / sparse / agentic / KG / causal). Score on depth · noise · compose. Discard if dominated. Continue if substrate. Saturation at iter 11 → synthesis selected Iter 2 → Iter 5 → Iter 12 as V1 pipeline.

~85min budget · 11 advanced · 1 discarded
UX research loop

Multiple persona walkthroughs (ACCA · SWE · Doctor) at every cycle that touched ranking. Score on category-fit · obvious-misses · niche-handling. Backtrack when persona X gets persona Y's tools. Pattern: don't just trust the algorithm — walk a persona by hand each time the inputs change.

3 personas · re-run on every algo change

the loop forces you to have a separate evaluator from the generator. you stop falling in love with your own first answer.

12 · hackathon learnings — ii

Three agents.
Three specialized loops.

agent 1 · MVP shipper

Worked the active SDD cycle. Stay in the spec-delta. Tick tasks as they finish. Commit per implementation step. Mostly Edit + Bash + Read tools.

opus · primary thread
agent 2 · research + eng

Spawned for self-contained sub-investigations: algorithm iterations · benchmark methodologies · "should we use Postgres or Mongo here." Returns a digest, doesn't pollute the main thread's context.

subagent · explore + general-purpose
agent 3 · discipline keeper

Watched for SDD/VKF compliance. Did I ingest before pasting? Did I /sdd:start before coding? Did the spec-delta have GWT? Pre-commit + CLAUDE.md guardrails.

CLAUDE.md routing · vkf/sdd commands

delegation isn't just speed — it's protecting the main loop's context window. agent 2's 30 reads don't enter agent 1's history.

what worked
  • Vertical slices, not horizontal layers. Each cycle shipped backend + frontend + tests for one user-visible thing. Never "build all backends, then all frontends."
  • Real data > placeholder. 27 seeded users · 546 real tool embeddings · the demo runs on live Weaviate + Mongo, not a fixture.
  • Loop discipline beats inspiration. Hardest decisions came from scoring then comparing, not from "what feels right."
what I'd change next time
  • Smaller cycles for UI. Three 1500-line cycles beat one 4000-line cycle for review fidelity.
  • Earlier deploy target. Local-only until day 5 means demo-day networking risk. Even a Fly.io + Vercel skeleton on day 2 is worth it.
  • Persona walkthrough as required validation. Tick was bypassed once or twice; a missed walkthrough caused the "marketer gets dev tools" bug.
