# Iteration 2 — LLM Intent / Friction Extraction as a Pre-Encoding Layer

**Question.** Iter 1 (raw-text embeddings + cosine over ~300 vendor capability blurbs) hit a friction-vs-feature vocabulary wall. Does an LLM pre-pass that extracts structured intent (frictions, desired capabilities, excluded tools, workflow edges, hypothetical wishes) before embedding actually move the needle? Literature scan below.

## Per-paper takeaways

1. **HyDE — Precise Zero-Shot Dense Retrieval without Relevance Labels (Gao et al., 2022).** LLM hallucinates a hypothetical "answer document" for the query, then embeds *that* and does cosine — the dense bottleneck filters fabricated details. Direct evidence that pivoting through an LLM-generated artifact beats embedding the raw user utterance, because the artifact already speaks corpus-language. ([abs](https://arxiv.org/abs/2212.10496))

2. **Query2Doc — Query Expansion with Large Language Models (Wang, Yang, Wei, EMNLP 2023).** Few-shot prompts an LLM to write a pseudo-doc, concatenates it with the query. +3–15% BM25 on MS-MARCO/TREC-DL, also lifts dense retrievers. Confirms HyDE-style expansion isn't a one-trick pony — works across sparse and dense, in-domain and out. ([abs](https://arxiv.org/abs/2303.07678))

3. **Query Expansion by Prompting LLMs (Jagerman et al., 2023).** Tests prompt families; chain-of-thought query expansion wins because it forces the LLM to *decompose* the query before expanding. The decomposition step is the part Mesh actually needs — frictions like "I lose track of standup promises" must be broken into capability sub-claims. ([abs](https://arxiv.org/abs/2305.03653))

4. **Doc2Query / docTTTTTquery (Nogueira et al., 2019).** The dual move: instead of bringing queries closer to docs, bring docs closer to plausible queries by prepending predicted queries to each document. Highly relevant — we can run the *same* LLM over each tool's vendor blurb to generate the friction/workflow utterances a user might phrase, then embed both sides on equal footing. ([abs](https://arxiv.org/abs/1904.08375))

5. **ILLUMINER — Instruction-tuned LLMs as Few-shot Intent Classifier and Slot Filler (Mirza et al., LREC-COLING 2024).** Reframes IC+SF as plain text generation; FLAN-T5-11B beats GPT-3.5 in-context by 11–32 pts on slot filling. Validates that structured intent extraction (the core of our pre-pass) is solid with modest models and few-shot prompts — we don't need GPT-4-class reasoning per query. ([abs](https://arxiv.org/abs/2403.17536))

6. **Zero-shot Slot Filling in the Age of LLMs for Dialogue Systems (Rana et al., COLING 2025).** Real-world dialogue has "abrupt topic shifts, interruptions, implicit references" — exactly the noise mode of Mesh signals. Knowledge-distilling from a teacher LLM into a small extractive model yields +26% F1 over vanilla LLM and +34% over off-the-shelf extractors, at production latency. Suggests a path once the prompt-only version is validated. ([abs](https://arxiv.org/abs/2411.18980))

7. **Learning to Rewrite Negation Queries in Product Search (Guo et al., COLING-Industry 2025).** Direct attack on the "tried Reflect, didn't stick" / "exclude tools like X" failure mode. Trains a rewriter to split a negation query into clean positive + exclusion components a vanilla retriever can act on, instead of patching the retriever. Pattern transfers cleanly: our LLM pass should emit `excluded_tools[]` and `excluded_paradigms[]` as first-class slots. ([pdf](https://aclanthology.org/2025.coling-industry.49.pdf))

8. **Hypothetical Documents or Knowledge Leakage? Rethinking LLM-based Query Expansion (Yoon et al., 2025).** Important caution: HyDE/Query2Doc gains on fact-verification benchmarks correlate strongly with the LLM having *memorized* the gold evidence. Where the corpus is private and unmemorized — i.e., Mesh's ~300 niche tools, many obscure — the gain is much smaller than benchmark numbers imply. We should expect a real but modest lift, and stress-test on long-tail tools. ([abs](https://arxiv.org/abs/2504.14175))

9. **Large Language Models for Intent-Driven Session Recommendations (Sun et al., SIGIR 2024).** Closest to Mesh in spirit: variable-cardinality user intents per session, LLM as the intent inference layer, prompt self-refinement loop. Confirms the explicit-intent-then-recommend split works in a recommender setting (not just IR), and that intents should not be assumed fixed-count. ([acm](https://dl.acm.org/doi/10.1145/3626772.3657688))

## Synthesis — what this layer does for Mesh

**The pipeline.** Raw signal → LLM extractor (system prompt: schema with `frictions[]`, `desired_capabilities[]`, `excluded_tools[]`, `excluded_paradigms[]`, `workflow_edges[]`, `counterfactual_wishes[]`, `prior_tool_history[]`) → embed each populated field separately → cosine against the corpus, where corpus side has been Doc2Query-style expanded into the same field schema (LLM rewrites each vendor blurb into "what frictions does this tool kill / what capabilities does it offer / what workflow shape does it support"). Final score = weighted sum of per-field cosines, with `excluded_*` fields applied as hard filters or negative similarity.

**Depth gained.** Iter 1 reached *topical* match (note-taking ~ note-taking). The intent layer reaches *capability-claim* depth: "I lose track of standup promises" → `friction: ephemeral_verbal_commitments_lost` → matches tools whose blurbs (LLM-rewritten) include `kills: action-item-capture-from-meetings`, even though neither side originally said the other's words. The big win is on the friction-to-feature axis specifically.

**New noise modes absorbed.** (i) **Negation/exclusion** — `excluded_tools` becomes structured, no longer fights the embedding. (ii) **Friction paraphrase** — "I lose track" / "things slip" / "I forget" all canonicalize to one slot. (iii) **Multi-entity workflows** — "Notion to Linear every Tuesday" survives as a structured edge instead of being averaged into a vague vector. (iv) **Counterfactuals / wishes** — HyDE-style: the LLM can hallucinate the *tool that would kill this friction*, embed that, retrieve the closest real one.

**Gap that remains.** The matcher is still *bag-of-fields cosine*. It cannot reason about *combinations* — that a tool covering capability A + capability B + paradigm-fit is better than one covering A+A+A. It treats `prior_tool_history` as a hint, not as a hard constraint propagated through the workflow graph (e.g., "if user already lives in Linear, prefer tools with Linear connectors"). And it has no reranker to distinguish two tools that look equally good on every field but differ on a soft fit signal (price tier, paradigm vibe, learning curve). That last one — the **paradigm-fit / soft-rejection** problem ("Reflect was rejected because daily-note paradigm felt heavy") — is the biggest residual.

## Suggested next direction: contrastive intent-level learning of paradigm/fit embeddings

Of the candidates listed, the right one is **contrastive intent learning** over a paradigm-fit auxiliary embedding — *not* a cross-encoder reranker (premature; we don't have click data yet), *not* a GNN (no graph yet), *not* RAG-with-reasoning (overkill for ranking 300 items).

**Why.** The remaining gap is qualitative fit, not topical relevance: two tools can pass the structured-field match equally yet one feels like a Reflect-redux to a user who just rejected Reflect. This is exactly what contrastive training on `(user_intent, accepted_tool, rejected_tool)` triples teaches an embedding space. We can bootstrap triples cheaply: for every signal that contains `excluded_tools` or "didn't stick because…", that *is* a labeled negative against a paradigm-similar tool. Train a small projection head on top of the structured-field embeddings so that distance reflects paradigm/fit, not just capability overlap. This composes with the iter-2 LLM extractor (no rework) and produces clean training signal once Mesh has any usage data, while letting us punt cross-encoders, ColBERT, and graphs to later iterations where they're justified by data volume.
